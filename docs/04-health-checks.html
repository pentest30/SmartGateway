CHAPTER 04

Health Checks & Probing

SmartGateway actively probes destination health every 10 seconds, automatically evicting unhealthy servers and re-admitting them when they recover.

1 Two Health Check Systems

SmartGateway uses two complementary systems to maintain cluster health:

Active Health Probing

HealthProbeService is a background service that runs every 10 seconds. It sends GET /health to every destination. If the response is not 2xx (or the request fails), the destination is marked unhealthy and removed from the load-balancing pool.

TTL-Based Expiry

TtlExpiryService is a background service that runs every 5 seconds. For destinations with a non-zero TtlSeconds, it checks if the last heartbeat was too long ago. Expired destinations are marked unhealthy.

What Gets Probed

For each destination in the database, the probe sends:

HTTP
GET {destination.Address}/health
Timeout: 5 seconds

For example, if a destination has address http://my-service:8080, the probe hits http://my-service:8080/health.

Success and Failure Criteria

ResultIsHealthyEffect on Routing
HTTP 200-299 responsetrueDestination stays in the load-balancing pool
HTTP 4xx or 5xx responsefalseDestination is removed from the pool
Connection timeout (5s)falseDestination is removed from the pool
DNS resolution failurefalseDestination is removed from the pool
Any exceptionfalseDestination is removed from the pool
How eviction works

The DatabaseProxyConfigProvider only includes destinations where IsHealthy = true in the YARP configuration. When it reloads, unhealthy destinations are simply absent from the cluster's destination list. YARP never sees them.

  ----    ------------    ---------    -------------

Eviction Flow (Step by Step)

  1. HealthProbeService sends GET /health to destination
  2. Probe fails (timeout, error, non-2xx)
  3. dest.IsHealthy = false is saved to SQL Server
  4. The Host logs: Destination {DestId} in {ClusterId} is now UNHEALTHY
  5. On next config reload, DatabaseProxyConfigProvider skips this destination
  6. YARP no longer routes traffic to the unhealthy destination

Recovery Flow (Step by Step)

  1. The upstream service comes back online and responds to /health
  2. HealthProbeService gets a 200 OK
  3. dest.IsHealthy = true is saved to SQL Server
  4. The Host logs: Destination {DestId} in {ClusterId} is now HEALTHY
  5. On next config reload, the destination reappears in the cluster
  6. YARP starts routing traffic to it again

4 Under the Hood: The Probe Method

C#
private async Task<bool> ProbeAsync(HttpClient client, string address, CancellationToken ct)
{
    try
    {
        var uri = address.TrimEnd('/') + "/health";
        var response = await client.GetAsync(uri, ct);
        return response.IsSuccessStatusCode;  // true if 2xx
    }
    catch
    {
        return false;  // Any exception = unhealthy
    }
}

Key implementation details:

  • Uses IHttpClientFactory for proper connection pooling
  • Probe timeout is 5 seconds (set on the HttpClient)
  • The loop interval is 10 seconds (set on the BackgroundService)
  • All destinations are probed sequentially within each cycle
  • A single failed cycle doesn't stop the service; errors are logged and the next cycle runs

5 TTL-Based Expiry: TtlExpiryService

For destinations registered with a non-zero TtlSeconds, the TtlExpiryService provides a second layer of health checking based on heartbeat freshness. See the Service Registry chapter for full details on heartbeats.

TTL Expiry Logic 
  TtlExpiryService (runs every 5 seconds)
           |
           v
  Query: SELECT * FROM Destinations
         WHERE TtlSeconds > 0
           AND IsHealthy = true
           AND DATEDIFF(SECOND, LastHeartbeat, GETUTCDATE()) > TtlSeconds
           |
           v
  For each expired destination:
    dest.IsHealthy = false
    Log: "Destination {DestId} in {ClusterId} expired (TTL: {Ttl}s)"
           |
           v
  SaveChangesAsync()

The check runs every 5 seconds. A destination is evicted when:

SQL (conceptual)
-- Destination is expired when:
TtlSeconds > 0                             -- TTL is configured
AND IsHealthy = true                     -- Currently healthy
AND DATEDIFF(SECOND, LastHeartbeat, NOW()) > TtlSeconds  -- Heartbeat expired

6 Implementing /health in Your Service

For active health probing to work, your upstream services must expose a GET /health endpoint. Here is a minimal example:

C#
// In your upstream service's Program.cs or a Controller
app.MapGet("/health", () =>
{
    // Return 200 if the service is healthy
    // Return 503 if it's not ready (e.g., DB connection down)
    return Results.Ok(new { status = "healthy", timestamp = DateTime.UtcNow });
});

Node.js
// Express.js example
app.get('/health', (req, res) => {
    res.json({ status: 'healthy', timestamp: new Date().toISOString() });
});
Important: The endpoint must be /health

SmartGateway appends /health to the destination address. If your service's address is http://my-service:8080, the probe will hit http://my-service:8080/health. Make sure this path returns a 2xx status code when the service is healthy.
