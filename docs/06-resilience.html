Resilience with Polly v8

Per-cluster retry, circuit breaker, and timeout pipelines powered by Polly v8. Configure via database, cached per cluster, rebuilt on config change.

Pipeline Architecture

Request ──► [ Retry ] ──► [ Circuit Breaker ] ──► [ Timeout ] ──► Upstream │ │ │ │ on failure │ if open │ if exceeded └─► retry N times └─► fail fast (503) └─► cancel + throw

Each cluster gets its own ResiliencePipeline built from the GatewayResiliencePolicy entity. Pipelines are cached by cluster ID and rebuilt when configuration changes.

Circuit Breaker States

Closed

Normal operation. Requests flow through. Failures are counted within the sampling window.

Open

Too many failures. All requests fail fast with 503. No traffic reaches upstream.

Half-Open

After break duration, one probe request is allowed. If it succeeds, circuit closes. If it fails, circuit re-opens.

failures < ratio break duration elapsed ┌──────────────────────┐ ┌──────────────────────────┐ │ │ │ │ ▼ │ ▼ │ CLOSED ──failures > ratio──► OPEN ──timer──► HALF-OPEN │ ▲ │ │ │ │ │ │ │ └──────── probe succeeds ──────────────────┘ │ │ └─ probe fails┘

Retry Strategies

Exponential Backoff

Delay doubles with each attempt: 200ms, 400ms, 800ms, 1600ms...

Attempt 1: wait 200ms, retry
Attempt 2: wait 400ms, retry
Attempt 3: wait 800ms, retry
→ Give up after 3 attempts

Linear Backoff

Delay stays constant: 200ms, 200ms, 200ms...

Attempt 1: wait 200ms, retry
Attempt 2: wait 200ms, retry
Attempt 3: wait 200ms, retry
→ Give up after 3 attempts

Pipeline Caching

ResiliencePipelineRegistry

Pipelines are expensive to build. The ResiliencePipelineRegistry caches them by cluster ID:

// Get or create pipeline for a cluster
var pipeline = registry.GetOrCreate(policy);

// When config changes, invalidate the cache
registry.Invalidate("orders-service");

// Clear all cached pipelines
registry.Clear();

The registry uses ConcurrentDictionary for thread-safe access.

What Gets Retried

Only HttpRequestException triggers retries. This covers:

  • Connection refused (upstream down)
  • DNS resolution failures
  • TCP connection timeouts

HTTP error responses (4xx, 5xx) do not trigger retries by default. To change this, modify the ShouldHandle predicate in ResiliencePipelineFactory.
