CHAPTER 03

Load Balancing Policies

SmartGateway provides three load-balancing strategies: YARP's built-in RoundRobin, a custom Weighted policy, and a custom LatencyAware policy that uses p95 latency tracking.

1 Three Policies at a Glance

RoundRobin

Equal Distribution

YARP's built-in policy. Cycles through all healthy destinations in order, giving each an equal share of traffic. Simple and effective when all servers have similar capacity.

Weighted

Proportional Distribution

Custom SmartGateway policy. Each destination has a configurable weight (0-1000). Traffic is distributed proportionally. Higher weight = more traffic. Stored in the database.

LatencyAware

Performance-Based Distribution

Custom SmartGateway policy. Tracks the p95 response latency for each destination using a circular buffer of 100 samples. Faster servers automatically get more traffic.

The load-balancing policy is set on the cluster level. Each cluster independently chooses its policy.

2 RoundRobin (Built-in)

RoundRobin Traffic Distribution 
  Requests:  1   2   3   4   5   6   7   8   9   ...

  Dest A:    *           *           *
  Dest B:        *           *           *
  Dest C:            *           *           *

  Each destination gets exactly 33.3% of traffic.

RoundRobin is YARP's default policy. It cycles through available destinations in order. No configuration beyond the policy name is needed.

Algorithm

  1. YARP maintains an internal counter per cluster
  2. For each request, it picks destinations[counter % count]
  3. Increments the counter atomically
  4. Only healthy destinations are in the pool

3 Weighted (Custom)

Weighted Traffic Distribution 
  Weights:  A=300   B=100   C=100
  Total: 500

  Out of 10 requests, approximately:
  Dest A (60%):  * * * * * *
  Dest B (20%):  * *
  Dest C (20%):  * *

The WeightedLoadBalancingPolicy is a custom SmartGateway policy that reads destination weights from the database via DatabaseDestinationWeightProvider.

Algorithm

C# (simplified)
// 1. Build cumulative weight array
int totalWeight = 0;
for (int i = 0; i < destinations.Count; i++)
{
    var weight = weightProvider.GetWeight(clusterId, destId);
    totalWeight += Math.Max(0, weight);
    cumulativeWeights[i] = totalWeight;
}

// 2. Pick random number in [0, totalWeight)
var target = Random.Next(totalWeight);

// 3. Find the destination whose cumulative range contains target
for (int i = 0; i < cumulativeWeights.Length; i++)
{
    if (target < cumulativeWeights[i])
        return destinations[i];
}

Each destination's Weight field (stored in GatewayDestination) controls its share. The default weight is 100.

DestinationWeightShare
server-a300300/500 = 60%
server-b100100/500 = 20%
server-c100100/500 = 20%

Visual traffic split:

A: 60%
B: 20%
C: 20%

How weights are stored

Each GatewayDestination entity has a Weight property (int, default 100). The DatabaseDestinationWeightProvider reads this value at routing time. To change a destination's weight, update it via the database or API and reload.

4 LatencyAware (Custom)

LatencyAware: Inverse-Latency Weighted Selection 
  LatencyTracker
  +-----------------------------------------+
  |  Circular Buffer (size: 100)            |
  |                                         |
  |  server-a: [12ms,14ms,11ms,15ms,...] p95 = 18ms  |
  |  server-b: [45ms,52ms,48ms,41ms,...] p95 = 55ms  |
  |  server-c: [25ms,28ms,24ms,30ms,...] p95 = 32ms  |
  +-----------------------------------------+
         |
         v
  Inverse-latency weights:
  server-a: 1/(18+1)  = 0.0526  --> 50% of traffic
  server-b: 1/(55+1)  = 0.0179  --> 17% of traffic
  server-c: 1/(32+1)  = 0.0303  --> 30% of traffic

The LatencyAwareLoadBalancingPolicy automatically routes more traffic to faster servers. It measures actual response latency using a thread-safe circular buffer of 100 samples per destination.

Algorithm

C# (simplified)
// 1. Get p95 latency for each destination from LatencyTracker
for (int i = 0; i < destinations.Count; i++)
{
    var p95 = tracker.GetP95(clusterId, destId);
    latencies[i] = (dest, p95.TotalMilliseconds);
}

// 2. If no latency data yet, pick randomly
if (!hasData) return destinations[Random.Next(count)];

// 3. Compute inverse-latency weights: 1/(p95 + 1)
//    Lower latency = higher weight = more traffic
for (int i = 0; i < latencies.Length; i++)
{
    weights[i] = 1.0 / (latencies[i].p95Ms + 1);
    totalWeight += weights[i];
}

// 4. Weighted random selection
var target = Random.NextDouble() * totalWeight;

The Circular Buffer

The LatencyTracker uses a fixed-size circular buffer (default: 100 entries) per destination. This means:

  • Memory-efficient: Only the last 100 latency samples are kept
  • Responsive: If a server slows down, the p95 rises quickly as new slow samples replace fast ones
  • Thread-safe: Protected by a lock per buffer instance
Circular Buffer for server-a (size 100) 
  Index: [0] [1] [2] [3] [4] ... [97] [98] [99]
  Data:  12  14  11  15  13  ...  16   14   18   (ms)
                                            ^
                                          head
  New sample arrives (9ms):
  Index: [0] [1] [2] [3] [4] ... [97] [98] [99]
  Data:  9   14  11  15  13  ...  16   14   18   (ms)
          ^
        head (wraps around)

  p95 = Sort all samples, take value at index ceil(0.95 * count) - 1

Traffic distribution adapts to latency:

A (18ms p95): 50%
C (32ms p95): 30%
B (55ms p95): 20%

Cold start behavior

When a destination has no latency data (cold start), the LatencyAware policy falls back to random selection. After enough requests fill the circular buffer, it switches to inverse-latency weighting. Destinations with no data get the average latency weight of those that do.

5 Switching a Cluster's Policy

You can change a cluster's load-balancing policy at any time through the Admin API:

Try it

# Switch from RoundRobin to Weighted
curl -X PUT http://localhost:5002/admin/api/clusters/my-api \
  -H "Content-Type: application/json" \
  -d '{"loadBalancing": "Weighted", "isActive": true}'

# Switch to LatencyAware
curl -X PUT http://localhost:5002/admin/api/clusters/my-api \
  -H "Content-Type: application/json" \
  -d '{"loadBalancing": "LatencyAware", "isActive": true}'

# Always reload after changes
curl -X POST http://localhost:5000/_admin/reload

6 Choosing the Right Policy

ScenarioRecommended PolicyWhy
All servers have identical hardware RoundRobin Equal distribution is optimal when capacity is uniform
Canary deployment (90/10 split) Weighted Set stable=900, canary=100 for a 90/10 traffic split
Mixed hardware (some fast, some slow) LatencyAware Automatically adapts to actual response performance
Blue/green deployment Weighted Set blue=0, green=100 to shift all traffic to green
Microservices with variable load LatencyAware Handles transient slowdowns by shifting traffic away
