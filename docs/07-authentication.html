API Key Authentication

SmartGateway protects routes with API key authentication. Keys are hashed with SHA-256 before storage, validated on every request through the X-Api-Key header, and managed via the Admin UI or direct database operations.

How It Works

API key authentication in SmartGateway follows a straightforward security model. The raw key is shown to the user exactly once at creation time, and only the SHA-256 hash is persisted. When a request arrives, the gateway hashes the incoming key and compares it against stored hashes.

Security by design: Because only the SHA-256 hash is stored in the database, a database leak never exposes raw API keys. This follows the same pattern used for password storage.

The GatewayApiKey Entity

The GatewayApiKey entity lives in SmartGateway.Core.Entities and maps to the ApiKeys table in SQL Server.

namespace SmartGateway.Core.Entities;

public class GatewayApiKey
{
    public int Id { get; set; }
    public string KeyHash { get; set; }     // SHA-256 hex of the raw key
    public string Name { get; set; }        // Human-readable label
    public string? Scopes { get; set; }    // Comma-separated scopes
    public bool IsActive { get; set; }     // false = revoked
    public DateTime CreatedAt { get; set; }
    public DateTime? LastUsedAt { get; set; }
}

SHA-256 Hashing Internals

The ApiKeyValidator class in SmartGateway.Host.Auth handles both hashing and database lookup. Here is the hashing method:

public static string ComputeHash(string input)
{
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(input));
    return Convert.ToHexStringLower(bytes);
}

// Example:
// Input:  "sgw_a1b2c3d4e5f6..."
// Output: "3f2a9c8b71e4d0f6..."  (64 hex characters)

The key format generated by the Admin UI follows the pattern:

sgw_{Guid.NewGuid():N}

// Example output: sgw_8f14e45fceea167a5a36dedd4bea2543
One-time display: The raw key is shown to the user exactly once when created. After that, only the hash exists in the database. If the raw key is lost, the only option is to create a new one.

Validation Flow

When a request hits a route with RequiresAuth = true, the gateway extracts the X-Api-Key header and runs the validation pipeline:

public async Task<bool> ValidateAsync(string rawKey)
{
    // 1. Hash the incoming raw key
    var hash = ComputeHash(rawKey);

    // 2. Query the database for matching active key
    await using var context = await _contextFactory.CreateDbContextAsync();
    var key = await context.ApiKeys
        .AsNoTracking()
        .FirstOrDefaultAsync(k => k.KeyHash == hash && k.IsActive);

    if (key == null)
        return false;  // No match or key revoked

    // 3. Update LastUsedAt timestamp
    await using var updateCtx = await _contextFactory.CreateDbContextAsync();
    var toUpdate = await updateCtx.ApiKeys.FindAsync(key.Id);
    if (toUpdate != null)
    {
        toUpdate.LastUsedAt = DateTime.UtcNow;
        await updateCtx.SaveChangesAsync();
    }

    return true;
}

Key details about this flow:

  • The validator uses IDbContextFactory to create short-lived contexts, avoiding concurrency issues.
  • The LastUsedAt update uses a separate context so the read query stays AsNoTracking for performance.
  • Only keys where IsActive == true pass validation -- revoked keys are rejected even if the hash matches.

Enabling Authentication on Routes

Authentication is controlled per-route via two fields on the GatewayRoute entity:

public bool RequiresAuth { get; set; } = false;
public string? AuthPolicyName { get; set; }

To protect a route, set RequiresAuth = true. You can do this via the REST API:

# Create a route that requires API key authentication
curl -X POST http://localhost:5100/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "orders-api",
    "clusterId": "orders-cluster",
    "pathPattern": "/api/orders/{**catch-all}",
    "methods": "GET,POST,PUT,DELETE",
    "requiresAuth": true,
    "authPolicyName": "ApiKey"
  }'

Routes with RequiresAuth = false (the default) are publicly accessible -- no key needed.

Creating Keys via the Admin UI

The Admin UI provides a complete key management interface at /auth.

Navigate to Auth & Keys

Open the Admin UI and click Auth & Keys in the sidebar. You will see a table of all existing API keys showing their hash prefix, name, scopes, creation date, last used time, and status.

Click "+ New API Key"

This opens a modal dialog with two fields:

  • Name -- a human-readable label (e.g., "Production webhook", "CI pipeline")
  • Scopes -- optional comma-separated scopes (e.g., "read,write")

Click "Generate key"

The system generates a raw key in the format sgw_{guid}, computes its SHA-256 hash, and stores only the hash. The raw key is displayed in a green banner at the top of the page:

New key created sgw_8f14e45fceea167a5a36dedd4bea2543

Copy now -- this won't be shown again

Copy and store the raw key

Save the key in a secrets manager (Azure Key Vault, HashiCorp Vault, etc.). Once you dismiss the banner, the raw key is gone forever.

Using API Keys in Requests

Pass the raw API key in the X-Api-Key header:

Successful request (valid key)

# Request with a valid API key
curl -v http://localhost:5000/api/orders \
  -H "X-Api-Key: sgw_8f14e45fceea167a5a36dedd4bea2543"

# Response: 200 OK
# {
#   "orders": [
#     { "id": 1, "product": "Widget A", "quantity": 5 },
#     { "id": 2, "product": "Widget B", "quantity": 3 }
#   ]
# }

Failed request (no key)

# Request without an API key to a protected route
curl -v http://localhost:5000/api/orders

# Response: 401 Unauthorized
# HTTP/1.1 401 Unauthorized
# Content-Length: 0
# Date: Sat, 05 Apr 2026 12:00:00 GMT

Failed request (invalid key)

# Request with a wrong or revoked key
curl -v http://localhost:5000/api/orders \
  -H "X-Api-Key: sgw_invalid_key_here"

# Response: 401 Unauthorized
# HTTP/1.1 401 Unauthorized
# Content-Length: 0

Public route (no auth required)

# Routes with RequiresAuth=false are open
curl http://localhost:5000/api/public/health

# Response: 200 OK (no key needed)
# { "status": "healthy" }

Key Lifecycle Management

ActionEffectReversible?
CreateGenerates raw key, stores SHA-256 hash, sets IsActive = true--
RevokeSets IsActive = false -- key immediately stops workingYes (re-activate in DB)
DeleteRemoves the key row entirely from the databaseNo

In the Admin UI, active keys show a active badge and a "Revoke" button. Revoked keys show revoked and a "Delete" button for permanent removal.

Tip: Prefer revoking over deleting. A revoked key leaves an audit trail showing it existed, while deletion removes all evidence. You can always delete a revoked key later.

Scopes

The Scopes field on GatewayApiKey stores a comma-separated list of permissions. While the base ApiKeyValidator checks only for existence and active status, you can extend it to match scopes against route-level requirements.

// Example scope values stored in the database:
"read"            // Read-only access
"read,write"      // Read and write access
"admin"           // Full admin access
"orders:read"     // Namespaced read access to orders

Scopes are displayed in the Admin UI as individual badges next to each key, making it easy to audit which keys have which permissions.

Database Schema

The ApiKeys table is registered in SmartGatewayDbContext and created via EF Core migrations:

    [Scopes]     NVARCHAR(MAX)  NULL,
    [IsActive]   BIT            NOT NULL DEFAULT 1,
    [CreatedAt]  DATETIME2      NOT NULL,
    [LastUsedAt] DATETIME2      NULL
);

Security Best Practices

1. Use HTTPS in production

API keys are sent as plaintext headers. Without TLS, they can be intercepted. Always terminate TLS before the gateway or use HTTPS end-to-end.

2. Rotate keys regularly

Create a new key, distribute it to consumers, then revoke the old one. The LastUsedAt field helps identify which keys are still in active use.

3. Use scopes for least privilege

Issue keys with the minimum required scopes. A webhook that only reads data should not have a write scope.

4. Monitor LastUsedAt

Keys that haven't been used in a long time are candidates for revocation. Check the "Last used" column in the Admin UI regularly.

5. Store raw keys in a secrets manager

Never store raw API keys in source code, config files, or unencrypted databases. Use Azure Key Vault, AWS Secrets Manager, or HashiCorp Vault.

Full Integration Example

Here is a complete end-to-end example: create a protected cluster, route, and API key, then make authenticated requests.

# Step 1: Create a cluster
curl -X POST http://localhost:5100/api/clusters \
  -H "Content-Type: application/json" \
  -d '{"clusterId": "payments-svc", "loadBalancing": "RoundRobin"}'

# Step 2: Add a destination
curl -X POST http://localhost:5100/api/services/register \
  -H "Content-Type: application/json" \
  -d '{
    "clusterId": "payments-svc",
    "destinationId": "payments-1",
    "address": "http://10.0.1.50:8080"
  }'

# Step 3: Create a protected route
curl -X POST http://localhost:5100/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "payments-route",
    "clusterId": "payments-svc",
    "pathPattern": "/api/payments/{**catch-all}",
    "requiresAuth": true
  }'

# Step 4: Create an API key via the Admin UI at /auth
# (Copy the generated key: sgw_...)

# Step 5: Test the protected route
curl http://localhost:5000/api/payments/charge \
  -H "X-Api-Key: sgw_8f14e45fceea167a5a36dedd4bea2543" \
  -H "Content-Type: application/json" \
  -d '{"amount": 99.99, "currency": "USD"}'

# 200 OK -- request proxied to payments-svc
