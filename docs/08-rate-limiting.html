Rate Limiting

SmartGateway supports three rate limiting strategies -- FixedWindow, SlidingWindow, and TokenBucket -- configured per-route as JSON. Protect your upstream services from abuse while providing clear 429 Too Many Requests responses to clients.

Three Strategies

SmartGateway uses the .NET System.Threading.RateLimiting library to implement three distinct algorithms, each suited to different traffic patterns.

FixedWindow
Resets the counter at fixed intervals. Simple and predictable. Allows bursts at window boundaries.

The RateLimitConfig Model

Rate limiting is configured per-route via a JSON string stored in the RateLimitConfig column of the GatewayRoute table. The JSON is deserialized into this C# model:

namespace SmartGateway.Host.RateLimiting;

public class RateLimitConfig
{
    public string? Policy { get; set; }              // "FixedWindow", "SlidingWindow", "TokenBucket"
    public int PermitLimit { get; set; }              // Max requests per window
    public int WindowSeconds { get; set; }            // Window duration in seconds
    public int SegmentsPerWindow { get; set; } = 4;   // SlidingWindow only
    public int TokenLimit { get; set; }              // TokenBucket max tokens
    public int TokensPerPeriod { get; set; }          // Tokens added each period
    public int ReplenishmentPeriodSeconds { get; set; } // Refill interval
}

How Policies Are Built

The RateLimitPolicyBuilder uses a simple switch expression to create the correct RateLimiter instance based on the policy name:

public static RateLimiter? Build(RateLimitConfig? config)
{
    if (config?.Policy == null)
        return null;

    return config.Policy switch
    {
        "FixedWindow" => new FixedWindowRateLimiter(new FixedWindowRateLimiterOptions
        {
            PermitLimit = config.PermitLimit,
            Window = TimeSpan.FromSeconds(config.WindowSeconds),
            QueueLimit = 0   // No queuing -- reject immediately
        }),
        "SlidingWindow" => new SlidingWindowRateLimiter(new SlidingWindowRateLimiterOptions
        {
            PermitLimit = config.PermitLimit,
            Window = TimeSpan.FromSeconds(config.WindowSeconds),
            SegmentsPerWindow = config.SegmentsPerWindow,
            QueueLimit = 0
        }),
        "TokenBucket" => new TokenBucketRateLimiter(new TokenBucketRateLimiterOptions
        {
            TokenLimit = config.TokenLimit,
            ReplenishmentPeriod = TimeSpan.FromSeconds(config.ReplenishmentPeriodSeconds),
            TokensPerPeriod = config.TokensPerPeriod,
            QueueLimit = 0
        }),
        _ => null
    };
}
QueueLimit = 0: All three strategies are configured with zero queuing. When the limit is exceeded, the request is rejected immediately with 429 rather than being queued. This prevents cascading latency in a gateway context.

Configuration Examples

The RateLimitConfig JSON is stored directly in the GatewayRoute.RateLimitConfig column. Here are examples for each strategy:

FixedWindow: 100 requests per minute

{
  "Policy": "FixedWindow",
  "PermitLimit": 100,
  "WindowSeconds": 60
}

Every 60 seconds the counter resets to zero. If a client sends 100 requests in the first 5 seconds, they are blocked for the remaining 55 seconds.

SlidingWindow: 200 requests per minute, 4 segments

{
  "Policy": "SlidingWindow",
  "PermitLimit": 200,
  "WindowSeconds": 60,
  "SegmentsPerWindow": 4
}

The 60-second window is divided into 4 segments of 15 seconds each. As each segment expires, its count is subtracted. This prevents the boundary burst problem of fixed windows.

TokenBucket: 10 tokens max, refill 2 every 10 seconds

{
  "Policy": "TokenBucket",
  "TokenLimit": 10,
  "TokensPerPeriod": 2,
  "ReplenishmentPeriodSeconds": 10
}

The bucket starts full (10 tokens). Each request consumes 1 token. Every 10 seconds, 2 tokens are added back up to the maximum of 10. This provides a steady average rate of 12 requests/minute with burst capacity.

Configuration via REST API

Set rate limiting when creating or updating a route through the SmartGateway API:

Create a rate-limited route

curl -X POST http://localhost:5100/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "public-api",
    "clusterId": "api-cluster",
    "pathPattern": "/api/public/{**catch-all}",
    "methods": "GET",
    "rateLimitConfig": "{\"Policy\":\"SlidingWindow\",\"PermitLimit\":200,\"WindowSeconds\":60,\"SegmentsPerWindow\":4}"
  }'
Note: The rateLimitConfig value is a JSON string within JSON. The inner JSON must be escaped when sent as part of the route payload.

Update rate limits on an existing route

curl -X PUT http://localhost:5100/api/routes/public-api \
  -H "Content-Type: application/json" \
  -d '{
    "rateLimitConfig": "{\"Policy\":\"TokenBucket\",\"TokenLimit\":20,\"TokensPerPeriod\":5,\"ReplenishmentPeriodSeconds\":10}"
  }'

Remove rate limiting from a route

curl -X PUT http://localhost:5100/api/routes/public-api \
  -H "Content-Type: application/json" \
  -d '{"rateLimitConfig": null}'

How Rate Limit Config Is Stored

Rate limiting configuration lives on the GatewayRoute entity as a nullable JSON string:

public class GatewayRoute
{
    public string RouteId { get; set; }
    public string ClusterId { get; set; }
    public string? PathPattern { get; set; }
    // ...other fields...
    public string? RateLimitConfig { get; set; }  // JSON blob, null = no limit
}

In the database, this column stores the raw JSON text:

-- Example data in the Routes table
SELECT RouteId, RateLimitConfig
FROM Routes
WHERE RateLimitConfig IS NOT NULL;

-- RouteId       | RateLimitConfig
-- public-api    | {"Policy":"SlidingWindow","PermitLimit":200,...}
-- webhook-in    | {"Policy":"TokenBucket","TokenLimit":10,...}

The 429 Too Many Requests Response

Key behaviors when a request is rejected:

  • The request is never forwarded to the upstream cluster.
  • A Retry-After header may be included indicating when the client can retry.
  • The response body is empty -- the status code carries the meaning.
  • Because QueueLimit = 0, rejection is instant with no added latency.

Global vs. Per-Route Limiting

Per-Route Limiting (default approach)

Each route has its own RateLimitConfig JSON. This means different routes can have different limits. An internal admin API might allow 1000 req/min while a public search endpoint allows 50 req/min.

// Route: internal-api   -->  1000 req/min FixedWindow
// Route: public-search  -->  50 req/min SlidingWindow
// Route: webhook-in     -->  TokenBucket 10 tokens
// Route: docs-static    -->  no rate limit (null)
Global Limiting (middleware approach)

For a gateway-wide rate limit, you can configure ASP.NET Core's built-in rate limiting middleware in Program.cs. This acts as a first layer before per-route policies are evaluated.

// In Program.cs (SmartGateway.Host)
builder.Services.AddRateLimiter(options =>
{
    options.AddFixedWindowLimiter("global", opt =>
    {
        opt.PermitLimit = 5000;
        opt.Window = TimeSpan.FromMinutes(1);
        opt.QueueLimit = 0;
    });
    options.RejectionStatusCode = 429;
});

// In the middleware pipeline
app.UseRateLimiter();

Layered approach: In production, use both. Set a high global limit (e.g., 5000 req/min) as a safety net, then fine-tune individual route limits based on your upstream service capacity.

Choosing the Right Strategy

Use FixedWindow when...
  • Simplicity matters more than precision
  • Upstream services have hard per-minute quotas
  • You are fine with potential bursts at window boundaries
Use SlidingWindow when...
  • You need smoother rate enforcement without boundary spikes
  • APIs serve end-user traffic where fairness matters
  • You want the best balance of accuracy and simplicity
Use TokenBucket when...
  • You want to allow controlled bursts (e.g., batch uploads)
  • Upstream services prefer a steady average rate with burst tolerance
  • You need fine-grained control over refill rate and burst capacity

Complete Working Example

Let's set up a sliding window rate limit on a public search endpoint and verify it works:

Create the route with rate limiting

curl -X POST http://localhost:5100/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "search-api",
    "clusterId": "search-cluster",
    "pathPattern": "/api/search/{**catch-all}",
    "methods": "GET",
    "rateLimitConfig": "{\"Policy\":\"SlidingWindow\",\"PermitLimit\":5,\"WindowSeconds\":60,\"SegmentsPerWindow\":4}"
  }'

Send requests up to the limit
