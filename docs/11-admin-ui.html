Chapter 11

Admin UI

SmartGateway ships with a full-featured Blazor Server admin interface for managing every aspect of the gateway. Built with a dark theme, it provides 11 pages covering clusters, routes, destinations, health, resilience, authentication, canary deployments, and audit logging.

All 11 Pages

The Admin UI is a Blazor Server application (SmartGateway.Admin) with interactive server-side rendering. Each page communicates directly with the SQL Server database via IDbContextFactory<SmartGatewayDbContext>.

Clusters (/clusters)

The Clusters page displays all upstream service groups in a responsive card grid (2 columns).

Each cluster card contains:

  • Header row: Cluster ID, load balancing policy, destination count, and a health badge (healthy, degraded, or inactive).
  • Destination list: Each destination shown as a row with a colored health dot, address, weight (w:100), and status badge.
  • Action footer: Activate/Deactivate toggle button and a red Delete button.

Creating a Cluster

Click "+ New Cluster" to open the creation modal with two fields:

  • Cluster ID: A unique identifier (e.g., orders-service)
  • Load balancing: Dropdown with options: RoundRobin, LeastRequests, PowerOfTwoChoices, Random, Weighted

Creating a Route

The route creation modal provides these fields:

  • Route ID -- unique identifier
  • Cluster -- dropdown populated from existing clusters
  • Path pattern -- YARP path match (e.g., /api/orders/{**catch-all})
  • Methods -- comma-separated HTTP methods (e.g., GET,POST,PUT)
  • Priority -- integer, lower values match first
  • Match header -- optional header-based routing (e.g., X-Tenant-Id)
  • Header value -- value to match against the header

Destinations (/destinations)

The Destinations page lists all upstream service instances in a table with health management controls.

Table Columns

Destination ID, Cluster, Address (monospace), Weight, Health (healthy / evicted), Last heartbeat (time), and action buttons.

Health Management

Each destination has inline actions:

  • Evict: Marks a healthy destination as unhealthy, removing it from the load balancer.
  • Re-admit: Marks an evicted destination as healthy again.
  • Delete: Permanently removes the destination from the database.

Summary Badges

At the top of the table, summary badges show total count, healthy count (green), and unhealthy count (red) at a glance.

Canary Deployments (/canary)

The Canary page is purpose-built for managing weighted traffic distribution during gradual rollouts.

Interface Elements

For each cluster with destinations, the page shows:

  • Cluster header: ID, load balancing policy, destination count, active status.
  • Destination rows: Each has a health dot, name, address, a range slider (0-100), current weight, and calculated percentage.
  • Traffic distribution bar: A stacked horizontal bar at the bottom showing proportional shares. Each segment is colored by health status (accent for healthy, red for evicted).
Real-time updates: Dragging a slider immediately saves the new weight to the database. All percentages and the distribution bar recalculate on the fly. No save button needed.

Weight Calculation

// Displayed percentage for each destination:
percentage = (destination.Weight / cluster.Destinations.Sum(d => d.Weight)) * 100

// Example: v1=100, v2=10 --> total=110
// v1: 100/110 = 90.9%
// v2:  10/110 =  9.1%

Health Monitor (/health)

The Health page provides a visual grid of all destination health states with summary statistics.

KPI Stats

Four stat cards show: Total destinations, Healthy count (green), Evicted count (red if > 0), and Health rate percentage.

Health Grid

Each destination is shown as a card-like node in a responsive grid. Each node displays:

  • Name: ClusterId / DestinationId format
  • Address: Monospace text showing the upstream URL
  • Status badge: healthy or evicted
  • Weight: Shown as w:100 in monospace

Unhealthy nodes have a red-tinted border and show a "Re-admit" button for manual recovery.

Policy Configuration Modal

The "+ Configure Policy" button opens a modal with fields for: Cluster (dropdown), Retry max attempts, Backoff type (Exponential/Linear), Retry delay (ms), Circuit break duration (ms), and Timeout (ms). If a policy already exists for the selected cluster, it is replaced.

Auth & Keys (/auth)

The Auth & Keys page is the API key management center for the gateway.

Key Table

Each key row shows:

  • Key ID: First 12 characters of the hash followed by "..." (full hash never shown)
  • Name: Human-readable label
  • Scopes: Displayed as individual gray badge pills
  • Created: Date in yyyy-MM-dd format
  • Last used: Time (HH:mm:ss) or "never"
  • Status: active or revoked
  • Actions: "Revoke" (for active) or "Delete" (for revoked)

Key Creation Flow

When you create a new key:

  1. Click "+ New API Key" to open the modal
  2. Enter a Name and optional comma-separated Scopes
  3. Click "Generate key"
  4. The system generates a raw key (sgw_{guid}), computes its SHA-256 hash, and saves only the hash
  5. A green banner appears at the top with the raw key and a warning: "Copy now -- this won't be shown again"

Audit Log (/audit)

The Audit Log page provides a complete changelog of every entity modification in the gateway, with powerful filtering.

Filter Bar

Two dropdown filters at the top allow narrowing results:

  • Entity type: All entities, Route, Cluster, or Destination
  • Action: All actions, CREATE, UPDATE, DELETE, REGISTER, or DEREGISTER

Selecting a filter immediately refreshes the log (using Blazor's @bind:after pattern).

Log Entry Format

Each audit entry displays as a row with:

  • Operation indicator: A colored dot (green for CREATE/REGISTER, blue for UPDATE, red for DELETE/DEREGISTER)
  • Entity: EntityType / EntityId (e.g., "Route / orders-api")
  • Action & details: The action verb plus a truncated JSON snippet of new values (max 80 characters)
  • Timestamp: Time in HH:mm:ss format

The log shows the 100 most recent entries matching the current filters, ordered newest first.

Audit Log Entity

public class GatewayAuditLog
{
    public int Id { get; set; }
    public string EntityType { get; set; }  // "Route", "Cluster", "Destination"
    public string EntityId { get; set; }    // The entity's primary key
    public string Action { get; set; }      // "CREATE", "UPDATE", "DELETE", etc.
    public string ChangedBy { get; set; }   // Actor identity
    public string? OldValues { get; set; }  // JSON snapshot before change
    public string? NewValues { get; set; }  // JSON snapshot after change
    public DateTime ChangedAt { get; set; }
}

How Modals Work

Most creation actions (new cluster, new route, new destination, new API key, new resilience policy) use a consistent modal pattern implemented in Blazor.

Modal Structure

<!-- Blazor modal pattern used across all pages -->
@if (_showModal)
{
    <div class="modal-overlay open" @onclick="OnOverlayClick">
        <div class="modal" @onclick:stopPropagation="true">
            <div class="modal-title">New Entity</div>

            <!-- Form fields -->
            <div class="form-row">
                <div class="form-group">
                    <div class="form-label">Field Name</div>
                    <input @bind="_field" class="form-input" />
                </div>
            </div>

            <!-- Action buttons -->
            <div class="modal-actions">
                <button class="btn btn-ghost">Cancel</button>
                <button class="btn btn-primary" @onclick="Submit">Create</button>
            </div>
        </div>
    </div>
}

Modal Behaviors

BehaviorImplementation
OpenSet _showModal = true (button click handler)
Close on overlay click@onclick="OnOverlayClick" sets _showModal = false
Prevent close on modal body click@onclick:stopPropagation="true" on the inner .modal div

Startup Configuration

// SmartGateway.Admin/Program.cs
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddRazorComponents()
    .AddInteractiveServerComponents();

builder.Services.AddDbContextFactory<SmartGatewayDbContext>(options =>
    options.UseSqlServer(
        builder.Configuration.GetConnectionString("SmartGateway")));

var app = builder.Build();

// Auto-migrate on startup
using (var scope = app.Services.CreateScope())
{
    var dbFactory = scope.ServiceProvider
        .GetRequiredService<IDbContextFactory<SmartGatewayDbContext>>();
    using var db = dbFactory.CreateDbContext();
    db.Database.Migrate();
}

app.UseAntiforgery();
app.MapStaticAssets();
app.MapRazorComponents<App>()
    .AddInteractiveServerRenderMode();

app.Run();
