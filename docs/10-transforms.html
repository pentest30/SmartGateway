Chapter 10

Transforms

SmartGateway transforms let you modify request headers, response headers, and URL paths as traffic flows through the gateway. Defined per-route in the database, transforms are applied by YARP's built-in transform pipeline with zero custom middleware.

How Transforms Work

Transforms are modifications applied to HTTP requests and responses as they pass through the gateway. SmartGateway stores transforms in the GatewayTransform table and converts them into YARP transform dictionaries when loading configuration.

The GatewayTransform Entity

Each transform is a row in the Transforms table, linked to a route via RouteId:

namespace SmartGateway.Core.Entities;

public class GatewayTransform
{
    public int Id { get; set; }
    public string RouteId { get; set; }     // FK to GatewayRoute
    public string Type { get; set; }        // "RequestHeader", "ResponseHeader", "PathPrefix"
    public string Key { get; set; }         // Header name or transform key
    public string? Value { get; set; }      // Header value, prefix, etc.
    public string Action { get; set; } = "Set";  // "Set", "Append", "Remove"

    public GatewayRoute? Route { get; set; }
}

Transform Types in Detail

1. Request Header Transforms

Modify headers on the request before it reaches the upstream service.

Set Set
Overwrites the header value, creating it if absent.
Type:   RequestHeader
Key:    X-Gateway-RequestId
Value:  {request-id}
Action: Set
Append Append
Adds a value to an existing header (multi-value).
Type:   RequestHeader
Key:    X-Forwarded-For
Value:  gateway-01
Action: Append
Remove Remove
Strips the header entirely from the request.
Type:   RequestHeader
Key:    Authorization
Action: Remove
Remove Remove
Strip cookies from internal service calls.
Type:   RequestHeader
Key:    Cookie
Action: Remove

2. Response Header Transforms

Modify headers on the response before it reaches the client.

Set CORS Set
Add CORS headers for browser clients.
Type:   ResponseHeader
Key:    Access-Control-Allow-Origin
Value:  *
Action: Set
Remove Server Remove
Hide upstream server identity.
Type:   ResponseHeader
Key:    Server
Action: Remove

3. Path Prefix Stripping

Remove a prefix from the request path before forwarding. This is essential when the gateway route uses a prefix like /api/v1/orders but the upstream expects just /orders.

// Transform definition
Type:   PathPrefix
Key:    PathRemovePrefix    // convention key
Value:  /api/v1             // the prefix to strip

// Effect:
// Client requests:   GET /api/v1/orders/123
// Upstream receives:  GET /orders/123

How YARP AddTransforms Works

SmartGateway's DatabaseProxyConfigProvider reads transforms from the database and converts them into YARP's transform dictionary format. Here is the conversion logic:

// Inside DatabaseProxyConfigProvider.LoadFromDatabase()

var transforms = new List<Dictionary<string, string>>();

foreach (var t in routeTransforms)
{
    var dict = new Dictionary<string, string>();

    switch (t.Type)
    {
        case "RequestHeader" when t.Action == "Set":
            dict["RequestHeader"] = t.Key;
            dict["Set"] = t.Value ?? "";
            break;

        case "RequestHeader" when t.Action == "Append":
            dict["RequestHeader"] = t.Key;
            dict["Append"] = t.Value ?? "";
            break;

        case "RequestHeader" when t.Action == "Remove":
            dict["RequestHeaderRemove"] = t.Key;
            break;

        case "ResponseHeader" when t.Action == "Set":
            dict["ResponseHeader"] = t.Key;
            dict["Set"] = t.Value ?? "";
            break;

        case "ResponseHeader" when t.Action == "Remove":
            dict["ResponseHeaderRemove"] = t.Key;
            break;

        case "PathPrefix":
            dict["PathRemovePrefix"] = t.Value ?? "";
            break;
    }

    if (dict.Count > 0)
        transforms.Add(dict);
}

Each dictionary becomes one transform entry in the YARP RouteConfig.Transforms list. YARP processes them in order during the proxy pipeline.

YARP transform format: YARP uses dictionaries of string key-value pairs. Each dictionary represents one transform operation. The key names (RequestHeader, Set, Append, etc.) are defined by YARP's built-in transform system.

Configuring Transforms in the Database

Transforms are stored as rows in the Transforms table. A single route can have multiple transforms. Here is what the data looks like for a well-configured route:

-- Example: transforms for the "orders-api" route

INSERT INTO Transforms (RouteId, Type, [Key], Value, Action) VALUES
  -- Inject a unique request ID for tracing
  ('orders-api', 'RequestHeader',  'X-Gateway-RequestId', '{request-id}', 'Set'),

  -- Strip the Authorization header (gateway already validated)
  ('orders-api', 'RequestHeader',  'Authorization',       NULL,           'Remove'),

  -- Add CORS headers on the response
  ('orders-api', 'ResponseHeader', 'Access-Control-Allow-Origin',  '*',  'Set'),
  ('orders-api', 'ResponseHeader', 'Access-Control-Allow-Methods', 'GET,POST,PUT,DELETE', 'Set'),
  ('orders-api', 'ResponseHeader', 'Access-Control-Allow-Headers', 'Content-Type,X-Api-Key', 'Set'),

  -- Hide upstream server identity
  ('orders-api', 'ResponseHeader', 'Server',                       NULL,  'Remove'),

  -- Strip /api/v1 prefix before forwarding
  ('orders-api', 'PathPrefix',     'PathRemovePrefix',             '/api/v1', 'Set');

Practical Examples

Example 1: Inject X-Gateway-RequestId for distributed tracing

Add a unique request identifier to every proxied request so upstream services can correlate logs:

TypeRequestHeader
KeyX-Gateway-RequestId
Valuegateway-{request-id}
ActionSet
# Client request (no X-Gateway-RequestId)
curl http://localhost:5000/api/orders

# What the upstream receives:
# GET /api/orders HTTP/1.1
# Host: 10.0.1.10:8080
# X-Gateway-RequestId: gateway-3f2a9c8b-71e4-4d0f-...

Example 2: Strip Authorization header after gateway validation

When the gateway handles API key auth, there is no need to forward credentials to upstream services. Remove the Authorization header to reduce the attack surface:

TypeRequestHeader
KeyAuthorization
Value(null -- not needed for Remove)
ActionRemove
# Client request with auth
curl http://localhost:5000/api/orders \
  -H "Authorization: Bearer eyJhbGciOi..."

# What the upstream receives:
# GET /api/orders HTTP/1.1
# Host: 10.0.1.10:8080
# (Authorization header is gone)

Example 3: Add CORS headers for browser clients

Instead of configuring CORS on every upstream service, handle it at the gateway level with response transforms:

-- Three response header transforms for complete CORS

-- 1. Allow all origins
Type: ResponseHeader, Key: Access-Control-Allow-Origin,  Value: *, Action: Set

-- 2. Allow common methods
Type: ResponseHeader, Key: Access-Control-Allow-Methods, Value: GET,POST,PUT,DELETE,OPTIONS, Action: Set

-- 3. Allow custom headers
Type: ResponseHeader, Key: Access-Control-Allow-Headers, Value: Content-Type,X-Api-Key,Authorization, Action: Set
# Client sees CORS headers in every response:
# HTTP/1.1 200 OK
# Access-Control-Allow-Origin: *
# Access-Control-Allow-Methods: GET,POST,PUT,DELETE,OPTIONS
# Access-Control-Allow-Headers: Content-Type,X-Api-Key,Authorization
# Content-Type: application/json

Example 4: Path prefix stripping for API versioning

The gateway serves /api/v2/orders/* but the upstream expects just /orders/*:

TypePathPrefix
KeyPathRemovePrefix
Value/api/v2
ActionSet
# Client requests:    GET /api/v2/orders/123/items
# Upstream receives:  GET /orders/123/items

# Client requests:    POST /api/v2/orders
# Upstream receives:  POST /orders
Path matching: Make sure the route's PathPattern matches the full prefixed path (e.g., /api/v2/orders/{**catch-all}) while the PathPrefix transform strips the /api/v2 portion.

Transform Execution Order

Transforms are applied in the order they are loaded from the database. Within a single request, the order is:

  1. Request header transforms are applied before the request is forwarded (Set, Append, Remove).
  2. Path prefix transforms modify the URL path before forwarding.
  3. The request is forwarded to the upstream service.
  4. Response header transforms are applied to the upstream's response before returning to the client.
Order matters for headers: If you both Set and Append the same header, the Set runs first (by database insertion order), then Append adds to it. Plan your transforms accordingly.

Complete Route with Transforms

Here is a full example creating a route and its transforms for a well-secured API endpoint:

-- 1. Create the route
INSERT INTO Routes (RouteId, ClusterId, PathPattern, Methods, RequiresAuth, IsActive)
VALUES ('payments-v2', 'payments-svc', '/api/v2/payments/{**catch-all}',
        'GET,POST', 1, 1);

-- 2. Add transforms
INSERT INTO Transforms (RouteId, Type, [Key], Value, Action) VALUES
  -- Tracing: inject request ID
  ('payments-v2', 'RequestHeader',  'X-Gateway-RequestId',            '{request-id}',               'Set'),
  -- Tracing: inject gateway identity
  ('payments-v2', 'RequestHeader',  'X-Forwarded-By',                 'smartgateway-01',             'Set'),
  -- Security: strip auth header (already validated at gateway)
  ('payments-v2', 'RequestHeader',  'Authorization',                  NULL,                          'Remove'),
  -- Security: strip cookies
  ('payments-v2', 'RequestHeader',  'Cookie',                          NULL,                          'Remove'),
  -- CORS: allow browser access
  ('payments-v2', 'ResponseHeader', 'Access-Control-Allow-Origin',    'https://app.example.com',     'Set'),
  -- Security: hide server identity
  ('payments-v2', 'ResponseHeader', 'Server',                          NULL,                          'Remove'),
  ('payments-v2', 'ResponseHeader', 'X-Powered-By',                   NULL,                          'Remove'),
  -- Routing: strip the /api/v2 prefix
  ('payments-v2', 'PathPrefix',     'PathRemovePrefix',               '/api/v2',                     'Set');

With this configuration, a client request like:

curl https://gateway.example.com/api/v2/payments/charge \
  -H "Authorization: Bearer eyJ..." \
  -H "Cookie: session=abc123" \
  -H "Content-Type: application/json" \
  -d '{"amount": 50.00}'

Is transformed into this request to the upstream:

# POST /payments/charge HTTP/1.1
# Host: 10.0.1.50:8080
# Content-Type: application/json
# X-Gateway-RequestId: 3f2a9c8b-71e4-4d0f-...
# X-Forwarded-By: smartgateway-01
# (Authorization: stripped)
# (Cookie: stripped)
# (Path: /api/v2 prefix removed)

And the response to the client includes:

# HTTP/1.1 200 OK
# Content-Type: application/json
# Access-Control-Allow-Origin: https://app.example.com
# (Server header: removed)
# (X-Powered-By header: removed)

Hot Reload of Transforms

Like all gateway configuration, transforms are loaded from the database whenever SignalReload() is called on the DatabaseProxyConfigProvider. This means you can add, modify, or remove transforms without restarting the gateway.

// Transforms are loaded as part of the normal config reload
var dbTransforms = context.Transforms
    .AsNoTracking()
    .ToList()
    .GroupBy(t => t.RouteId)
    .ToDictionary(g => g.Key, g => g.ToList());

// Each route's transforms are then converted to YARP format
// and included in the RouteConfig.Transforms list
No downtime: Adding a transform to the database and triggering a config reload applies it to all subsequent requests. In-flight requests are not affected -- they complete with the old configuration.
