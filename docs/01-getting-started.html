1 Prerequisites

⚙

.NET 10 SDK

SmartGateway targets .NET 10. Download from dotnet.microsoft.com. Verify with dotnet --version.

🐳

Docker Desktop

Required for SQL Server. Ensure Docker is running on your machine before starting the database container.

🔌

curl (or similar)

All examples use curl. On Windows, Git Bash or PowerShell 7+ provide it. Postman also works.

Tip: Verify your tools

Run dotnet --version, docker --version, and curl --version to confirm everything is installed.

2 Start SQL Server

SmartGateway stores all configuration in SQL Server. The easiest way to get it running is with Docker Compose, which is already configured in the repository.

bash
# From the project root
docker compose up -d sqlserver

# Verify it's running
docker compose ps

This starts SQL Server 2022 on localhost:1433 with the following credentials:

ParameterValue
Host127.0.0.1,1433
Usersa
PasswordYourStrong@Passw0rd
DatabaseSmartGateway (auto-created on first run)
Note: Database auto-migration

SmartGateway.Host runs db.Database.Migrate() on startup, so the schema is created automatically. No manual SQL scripts needed.

3 Run the Three Projects

SmartGateway is composed of three independently runnable services. Open three terminal windows:

Terminal 1: Gateway Host (YARP Proxy)

bash
cd src/SmartGateway.Host
dotnet run
# Listening on http://localhost:5000
# This is where client traffic flows through

Terminal 2: Admin API

bash
cd src/SmartGateway.Api
dotnet run
# Listening on http://localhost:5002
# REST endpoints for managing clusters, routes, destinations

Terminal 3: Admin UI (Optional)

bash
cd src/SmartGateway.Admin
dotnet run
# Listening on http://localhost:5001
# Blazor Server dashboard
Alternative: Docker Compose for everything

Run docker compose up from the project root to start SQL Server and all three services together.

4 Create Your First Cluster, Route, and Destination

With all services running, let's wire up a complete proxy configuration. We'll create a cluster, add a destination (upstream server), create a route, and then trigger a YARP config reload.

Step 4a: Create a Cluster

A cluster groups one or more upstream servers. It also defines the load-balancing policy.

Try it

Create a cluster named "my-api" with RoundRobin load balancing:

curl -X POST http://localhost:5002/admin/api/clusters \
  -H "Content-Type: application/json" \
  -d '{
    "clusterId": "my-api",
    "loadBalancing": "RoundRobin",
    "isActive": true
  }'

Step 4b: Add a Destination

A destination is an upstream server address that belongs to a cluster. We add it via the service registry.

Try it

Register an upstream server at https://httpbin.org:

curl -X POST http://localhost:5002/admin/api/services/register \
  -H "Content-Type: application/json" \
  -d '{
    "clusterId": "my-api",
    "destinationId": "httpbin-1",
    "address": "https://httpbin.org",
    "ttlSeconds": 0
  }'

Setting ttlSeconds to 0 means this destination never expires (no heartbeat required).

Step 4c: Create a Route

A route maps incoming requests to a cluster. The path pattern /api/{**catch-all} forwards all requests under /api/ to the cluster.

Try it

curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "api-route",
    "clusterId": "my-api",
    "pathPattern": "/api/{**catch-all}",
    "isActive": true,
    "priority": 0
  }'

Step 4d: Reload YARP Config

After making changes through the Admin API, tell the Host to pick up the new configuration from the database:

Try it

curl -X POST http://localhost:5000/_admin/reload

You should see a response like: {"routes":1,"clusters":1}

5 Test Through the Proxy

Now the exciting part! Send a request through SmartGateway and watch it get proxied to your upstream server.

Try it

Send a request through the gateway (port 5000) that matches the /api/{**catch-all} route:

# This hits SmartGateway.Host on :5000
# It matches the /api/* route, which forwards to the "my-api" cluster
# The cluster routes to httpbin.org
curl http://localhost:5000/api/get

# Try with headers to see them forwarded
curl http://localhost:5000/api/headers \
  -H "X-Custom-Header: Hello from SmartGateway"

# POST request
curl -X POST http://localhost:5000/api/post \
  -H "Content-Type: application/json" \
  -d '{"message": "proxied through SmartGateway"}'

6 What Just Happened?

Let's trace the full request lifecycle:

Request Flow 
  curl :5000/api/get
         |
         v
  +----------------+     +-----------------+
  | Route Matching |     | SQL Server      |
  | Path: /api/*   |<----|  GatewayRoute   |
  +-------+--------+     |  GatewayCluster |
          |               |  GatewayDest.   |
          v               +-----------------+
  +----------------+
  | Load Balancer  |
  | Policy:        |
  |  RoundRobin    |
  +-------+--------+
          |
          v
  +----------------+
  | Proxy Forward  |
  | -> httpbin.org |
  +-------+--------+
          |
          v
  Response returned
  to client
  1. Client sends request to http://localhost:5000/api/get
  2. YARP matches the route — the path /api/get matches the pattern /api/{**catch-all}
  3. Route resolves to cluster — the route's clusterId is "my-api"
  4. Load balancer picks a destination — with one destination, RoundRobin selects httpbin-1
  5. Request is forwarded — YARP proxies the request to https://httpbin.org/api/get
  6. Response flows back through SmartGateway to the client

7 Verify Your Configuration

Use the Admin API to inspect what you've created:

Try it

# List all clusters (with their destinations)
curl http://localhost:5002/admin/api/clusters | python3 -m json.tool

# List all routes
curl http://localhost:5002/admin/api/routes | python3 -m json.tool

# List all registered services/destinations
curl http://localhost:5002/admin/api/services | python3 -m json.tool

8 Running the Test Suite

SmartGateway has comprehensive tests covering unit, integration, and end-to-end proxy scenarios:

bash
# Run all 137 tests
dotnet test SmartGateway.sln

# Run only unit tests
dotnet test tests/SmartGateway.Core.Tests
dotnet test tests/SmartGateway.Host.Tests

# Run integration tests (requires Docker for Testcontainers)
dotnet test tests/SmartGateway.Integration.Tests
