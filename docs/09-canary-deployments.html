Canary Deployments

Roll out new service versions gradually using weighted destinations. SmartGateway's Weight field on each destination controls traffic distribution, letting you shift from 0% to 100% with real-time monitoring via the Admin UI's Canary page.

What Is a Canary Deployment?

A canary deployment sends a small percentage of traffic to a new version of a service while the majority continues to hit the stable version. If the new version performs well (low errors, acceptable latency), you gradually increase its share. If problems appear, you roll back by setting its weight to zero.

In SmartGateway, canary deployments are managed through the Weight property on GatewayDestination. When a cluster uses the Weighted load balancing policy, traffic is distributed proportionally to each destination's weight.

The Weight Model

Each GatewayDestination has a Weight property (integer, default 100, range 0-100):

public class GatewayDestination
{
    public int Id { get; set; }
    public string ClusterId { get; set; }
    public string DestinationId { get; set; }
    public string Address { get; set; }
    public bool IsHealthy { get; set; } = true;
    public int Weight { get; set; } = 100;          // 0-100, controls traffic share
    public DateTime? LastHeartbeat { get; set; }
    public int TtlSeconds { get; set; } = 0;
}

Traffic distribution is calculated as:

// Traffic percentage for destination X:
percentage = (weight_x / sum_of_all_weights) * 100

// Example with two destinations:
// v1 weight=100, v2 weight=10  -->  total=110
// v1 gets 100/110 = 90.9% of traffic
// v2 gets  10/110 =  9.1% of traffic

Step-by-Step Canary Rollout

Ensure the cluster uses Weighted load balancing

Your cluster must use the Weighted load balancing policy for weight-based distribution to take effect.

curl -X POST http://localhost:5100/api/clusters \
  -H "Content-Type: application/json" \
  -d '{
    "clusterId": "orders-svc",
    "loadBalancing": "Weighted"
  }'
Existing cluster? Update the load balancing policy to Weighted via the Clusters page in the Admin UI or through the API.

Verify the stable v1 destination exists

The existing production destination should already be running with weight 100:

# Existing destination: orders-v1 at weight 100
curl http://localhost:5100/api/services/orders-svc

# Response:
# [
#   {
#     "destinationId": "orders-v1",
#     "address": "http://10.0.1.10:8080",
#     "weight": 100,
#     "isHealthy": true
#   }
# ]

Register the canary v2 destination with low weight

Add the new version with a weight of 10 (about 9% of traffic):

curl -X POST http://localhost:5100/api/services/register \
  -H "Content-Type: application/json" \
  -d '{
    "clusterId": "orders-svc",
    "destinationId": "orders-v2",
    "address": "http://10.0.1.20:8080",
    "weight": 10
  }'

Monitor in the Admin UI Canary page

Navigate to /canary in the Admin UI. You will see:

  • Each cluster with its destinations listed
  • A range slider (0-100) for each destination's weight
  • Real-time percentage calculations
  • A stacked traffic distribution bar showing the visual split
  • Health indicators (active / evicted) for each destination

Gradually increase v2 weight

If v2 is healthy and performing well, use the slider or API to increase its weight:

T+0 (initial deploy)
v1=100, v2=10 — 9.1% to canary
T+15min (looking good)
v1=100, v2=25 — 20% to canary
T+30min (gaining confidence)
v1=50, v2=50 — 50/50 split
T+1hr (full rollout)
v1=0, v2=100 — 100% to new version

Decommission v1

Once v2 is fully rolled out, remove the old destination:

# Remove the old v1 destination
curl -X POST http://localhost:5100/api/services/deregister \
  -H "Content-Type: application/json" \
  -d '{"clusterId": "orders-svc", "destinationId": "orders-v1"}'

Weight of 0: Setting a destination's weight to 0 effectively removes it from the load balancer without deleting it. This is useful for instant rollback -- just set v2's weight back to 0.

Instant Rollback

If the canary shows problems (errors, high latency), roll back instantly by setting the v2 weight to zero:

# Emergency rollback: set v2 weight to 0
# Use the Canary page slider or the API:
curl -X PUT http://localhost:5100/api/destinations/orders-v2 \
  -H "Content-Type: application/json" \
  -d '{"weight": 0}'

# All traffic immediately returns to v1
Why not delete? Setting weight to 0 is better than deleting the destination because: (1) you preserve the configuration for later retry, (2) the audit log records a weight change rather than a deletion, and (3) you can resume the rollout by increasing the weight again.

The Admin UI Canary Page

The Canary page at /canary provides a visual interface for managing weighted deployments.

Page Layout

Each cluster with destinations is displayed as a card containing:

  • Cluster header: Shows the cluster ID, load balancing policy, destination count, and active status badge.
  • Destination rows: Each destination has a health indicator dot, name, address, a range slider (0-100), the current weight value, and the calculated percentage.
  • Traffic distribution bar: A horizontal stacked bar at the bottom showing each destination's proportional share. Healthy destinations show in accent color, evicted ones in red.
  • Total weight: Displayed as a monospace value at the far right of the bar.
Using the Sliders

Drag any destination's slider to change its weight from 0 to 100. The change is saved immediately to the database when you release the slider. All percentages and the traffic distribution bar recalculate instantly. No page refresh or "Save" button is needed.

How Weighted Load Balancing Works

SmartGateway implements weighted load balancing through the WeightedLoadBalancingPolicy which reads weights from the database via IDestinationWeightProvider:

// Simplified flow inside WeightedLoadBalancingPolicy

// 1. Get all healthy destinations for the cluster
var destinations = availableDestinations;

// 2. Read weights from the database via IDestinationWeightProvider
var weights = await _weightProvider.GetWeightsAsync(clusterId);

// 3. Build a weighted selection
//    Random number [0, totalWeight) picks the destination
var totalWeight = destinations.Sum(d => weights[d.DestinationId]);
var random = Random.Shared.Next(totalWeight);

var cumulative = 0;
foreach (var dest in destinations)
{
    cumulative += weights[dest.DestinationId];
    if (random < cumulative)
        return dest;  // Selected!
}
Health-aware: The weighted load balancer only considers destinations where IsHealthy = true. If v2 becomes unhealthy, all its traffic automatically routes to v1 regardless of weights.

Canary Best Practices

1. Start small

Begin with weight 1-10 for the canary (1-9% traffic). This limits blast radius while still generating enough traffic for meaningful metrics.

2. Monitor before increasing

Wait at least 10-15 minutes at each weight level. Check error rates, latency percentiles, and log output before increasing the canary's share.

3. Use health checks

Configure active health probes on the canary destination. If v2 starts failing health checks, it will be automatically evicted -- an automatic rollback.

4. Keep v1 around

Don't delete v1 until v2 has been running at 100% weight for a meaningful period (hours or days). Set v1's weight to 0 but keep it registered for fast rollback.

5. Audit trail

All weight changes are logged in the Audit Log. Review /audit to track who changed what weights and when -- essential for incident post-mortems.
