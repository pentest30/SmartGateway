CHAPTER 02

Routing & Clusters

Learn how YARP matches incoming requests to clusters, and how to configure path patterns, host matching, method filtering, header matching, and priority ordering.

1 Core Concepts

How Route Matching Works in YARP 
  Incoming Request
  GET /api/users HTTP/1.1
  Host: api.example.com
  X-Version: v2
         |
         v
  +-------------------------------+
  | YARP Route Matching Engine     |
  |                               |
  |  For each route (by Priority):|
  |  1. Check Path pattern        |
  |  2. Check Hosts list          |
  |  3. Check Methods list        |
  |  4. Check Headers match       |
  |                               |
  |  First match wins!            |
  +-------------------------------+
         |
         v
  +-------------------------------+
  |  Matched Route: "api-v2"      |
  |  -> ClusterId: "backend-v2"  |
  +-------------------------------+
         |
         v
  Forward to cluster destinations

In SmartGateway, routing is stored in the GatewayRoute table and loaded into YARP's configuration by DatabaseProxyConfigProvider. Each route has these matching criteria:

FieldTypeDescription
RouteIdstringUnique identifier for the route
ClusterIdstringWhich cluster to forward matching requests to
PathPatternstringURL path pattern (supports {**catch-all})
HostsstringComma-separated list of hostnames
MethodsstringComma-separated HTTP methods (GET, POST, etc.)
MatchHeaderstringHeader name to match on
MatchHeaderValuestringRequired header value
PriorityintLower number = higher priority (evaluated first)
IsActiveboolOnly active routes are loaded into YARP

2 Path Pattern Matching

YARP uses ASP.NET Core's route template syntax. The most powerful pattern is the catch-all parameter {**catch-all}, which captures the remainder of the path.

Catch-All Pattern

The pattern /api/{**catch-all} matches any path that starts with /api/:

PatternRequest PathMatch?
/api/{**catch-all}/api/usersYes
/api/{**catch-all}/api/users/123/ordersYes
/api/{**catch-all}/apiYes
/api/{**catch-all}/docs/introNo

Try it

Create a catch-all route for an API backend:

curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "catch-all-api",
    "clusterId": "my-api",
    "pathPattern": "/api/{**catch-all}",
    "priority": 10,
    "isActive": true
  }'

Exact Path Pattern

Without {**catch-all}, the pattern matches exactly:

PatternRequest PathMatch?
/health/healthYes
/health/health/checkNo

Try it

Create an exact-path route for a health endpoint:

curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "health-exact",
    "clusterId": "my-api",
    "pathPattern": "/health",
    "priority": 1,
    "isActive": true
  }'

3 Host Matching

Routes can be restricted to specific hostnames. This enables virtual hosting where the same gateway serves different backends based on the domain name.

Host-Based Routing 
  api.example.com  ------>  Route "api-route"   -->  Cluster: api-backend
  web.example.com  ------>  Route "web-route"   -->  Cluster: web-backend
  docs.example.com ------>  Route "docs-route"  -->  Cluster: docs-backend

The Hosts field is a comma-separated list. If set, YARP only matches the route when the Host header matches one of the listed hostnames.

Try it

Create a host-specific route:

curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "api-host-route",
    "clusterId": "api-backend",
    "pathPattern": "/{**catch-all}",
    "hosts": "api.example.com,api.example.io",
    "priority": 5,
    "isActive": true
  }'

# Don't forget to reload YARP
curl -X POST http://localhost:5000/_admin/reload

# Test with a Host header
curl http://localhost:5000/users \
  -H "Host: api.example.com"
How YARP parses Hosts

SmartGateway stores hosts as a comma-separated string. The DatabaseProxyConfigProvider splits it using Hosts.Split(',', StringSplitOptions.TrimEntries) and passes it as a string array to YARP's RouteMatch.Hosts.

4 Method Filtering

Restrict a route to specific HTTP methods. This is useful for separating read and write traffic to different clusters.

Method-Based Routing 
  GET  /orders/{**catch-all}  -->  Route "orders-read"   -->  Cluster: read-replicas
  POST /orders/{**catch-all}  -->  Route "orders-write"  -->  Cluster: primary-db
  PUT  /orders/{**catch-all}  -->  Route "orders-write"  -->  Cluster: primary-db

Try it

Create separate read and write routes:

# Read route - only GET requests
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "orders-read",
    "clusterId": "read-replicas",
    "pathPattern": "/orders/{**catch-all}",
    "methods": "GET",
    "priority": 3,
    "isActive": true
  }'

# Write route - POST, PUT, DELETE
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "orders-write",
    "clusterId": "primary-db",
    "pathPattern": "/orders/{**catch-all}",
    "methods": "POST,PUT,DELETE",
    "priority": 3,
    "isActive": true
  }'

# Reload and test
curl -X POST http://localhost:5000/_admin/reload

# This hits read-replicas cluster
curl http://localhost:5000/orders/123

# This hits primary-db cluster
curl -X POST http://localhost:5000/orders \
  -H "Content-Type: application/json" \
  -d '{"item": "widget", "quantity": 5}'

5 Header Matching

Routes can match on a specific HTTP header and its value. This is powerful for API versioning, canary releases, and A/B testing.

Header-Based Routing (API Versioning) 
  X-Version: v1  -->  Route "api-v1"  -->  Cluster: backend-v1
  X-Version: v2  -->  Route "api-v2"  -->  Cluster: backend-v2
  (no header)    -->  Route "api-default" --> Cluster: backend-v2  (catch-all)

SmartGateway uses two fields: matchHeader (the header name) and matchHeaderValue (the required value). The DatabaseProxyConfigProvider converts these into YARP's RouteHeader matching.

Try it

Implement API versioning via headers:

# V1 route - matches when X-Version: v1
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "api-v1",
    "clusterId": "backend-v1",
    "pathPattern": "/api/{**catch-all}",
    "matchHeader": "X-Version",
    "matchHeaderValue": "v1",
    "priority": 1,
    "isActive": true
  }'

# V2 route - matches when X-Version: v2
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "api-v2",
    "clusterId": "backend-v2",
    "pathPattern": "/api/{**catch-all}",
    "matchHeader": "X-Version",
    "matchHeaderValue": "v2",
    "priority": 1,
    "isActive": true
  }'

# Default route - no header restriction (lower priority)
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "api-default",
    "clusterId": "backend-v2",
    "pathPattern": "/api/{**catch-all}",
    "priority": 10,
    "isActive": true
  }'

curl -X POST http://localhost:5000/_admin/reload

# Test: hits backend-v1
curl http://localhost:5000/api/users -H "X-Version: v1"

# Test: hits backend-v2
curl http://localhost:5000/api/users -H "X-Version: v2"

# Test: no header, falls through to default (backend-v2)
curl http://localhost:5000/api/users

6 Priority Ordering

When multiple routes could match a request, priority determines which one wins. Lower numbers mean higher priority (evaluated first).

The Priority Rule

YARP evaluates routes in ascending Priority order. The first route whose criteria match the incoming request is selected. All remaining routes are skipped.

Priority Evaluation Order 
  Incoming: GET /api/admin/dashboard

  Priority 0: Path=/api/admin/{**catch-all}  Host=*  Methods=*   <-- MATCH! Stop here.
  Priority 5: Path=/api/{**catch-all}        Host=*  Methods=GET   (never evaluated)
  Priority 10: Path=/{**catch-all}            Host=*  Methods=*     (never evaluated)

A practical priority strategy:

PriorityUse CaseExample
0Exact paths / special endpoints/health, /api/admin/*
1-4Header/method-specific routesAPI versioning, read vs. write split
5-9Host-specific routesVirtual hosting by domain
10+Catch-all / fallback routes/{**catch-all}

Try it

Create a specific admin route (priority 0) alongside a general catch-all (priority 10):

# High-priority admin route
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "admin-panel",
    "clusterId": "admin-backend",
    "pathPattern": "/api/admin/{**catch-all}",
    "priority": 0,
    "isActive": true
  }'

# Low-priority catch-all
curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "general-api",
    "clusterId": "general-backend",
    "pathPattern": "/api/{**catch-all}",
    "priority": 10,
    "isActive": true
  }'

curl -X POST http://localhost:5000/_admin/reload

# Hits admin-backend (priority 0 wins)
curl http://localhost:5000/api/admin/dashboard

# Hits general-backend (no higher priority match)
curl http://localhost:5000/api/users

7 Combining Multiple Criteria

A single route can combine path, host, method, and header matching. All specified criteria must match (logical AND) for the route to be selected.

Try it

Create a route that combines path + host + method + header:

curl -X POST http://localhost:5002/admin/api/routes \
  -H "Content-Type: application/json" \
  -d '{
    "routeId": "premium-write-v2",
    "clusterId": "premium-backend",
    "pathPattern": "/api/{**catch-all}",
    "hosts": "premium.example.com",
    "methods": "POST,PUT",
    "matchHeader": "X-Version",
    "matchHeaderValue": "v2",
    "priority": 0,
    "isActive": true
  }'

curl -X POST http://localhost:5000/_admin/reload

# This matches all four criteria
curl -X POST http://localhost:5000/api/orders \
  -H "Host: premium.example.com" \
  -H "X-Version: v2" \
  -H "Content-Type: application/json" \
  -d '{"item": "gold-plan"}'

8 Managing Routes via the API

Full CRUD operations are available through the Admin API at /admin/api/routes:

Try it

# List all routes (ordered by priority)
curl http://localhost:5002/admin/api/routes

# Update a route (change priority and path)
curl -X PUT http://localhost:5002/admin/api/routes/api-v1 \
  -H "Content-Type: application/json" \
  -d '{
    "pathPattern": "/v1/api/{**catch-all}",
    "clusterId": "backend-v1",
    "priority": 2,
    "isActive": true
  }'

# Disable a route (soft delete)
curl -X PUT http://localhost:5002/admin/api/routes/api-v1 \
  -H "Content-Type: application/json" \
  -d '{
    "pathPattern": "/v1/api/{**catch-all}",
    "clusterId": "backend-v1",
    "isActive": false
  }'

# Delete a route permanently
curl -X DELETE http://localhost:5002/admin/api/routes/api-v1

# Always reload after changes!
curl -X POST http://localhost:5000/_admin/reload
Remember: Always reload after changes

The Admin API writes to the database, but the Host picks up changes only when you call POST /_admin/reload. This is by design: it allows you to batch multiple changes and reload once.
